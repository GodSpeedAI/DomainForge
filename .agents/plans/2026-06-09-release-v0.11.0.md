# DomainForge v0.11.0 Release Plan

> **For Codex:** REQUIRED SUB-SKILL: Use superpowers:verification-before-completion before claiming the release is complete.

**Goal:** Prepare and publish the stable v0.11.0 semantic release for DomainForge.

**Architecture:** Rust `sea-core` remains canonical. Python, TypeScript napi, and WASM bindings ship in lockstep through package metadata updates and GitHub Release publish workflows.

**Tech Stack:** Rust/Cargo, PyO3/maturin, napi-rs/npm, wasm-pack, GitHub Actions, `gh` CLI.

## Tasks

### Task 1: Release Readiness

**Files:**
- Read: `.github/copilot-instructions.md`
- Read: `AGENTS.md`
- Read: `.agents/current_state.md`
- Read: `.agents/next_steps.md`
- Read: `docs/RELEASE_PROCESS.md`
- Read: `docs/playbooks/local-release-preparation.md`

**Steps:**
1. Confirm the working branch is clean and up to date.
2. Confirm `v0.11.0` does not already exist locally, remotely, or on package registries.
3. Confirm GitHub CLI authentication is available for tag/release publishing.

### Task 2: Version Bump

**Files:**
- Modify: `sea-core/Cargo.toml`
- Modify: `pyproject.toml`
- Modify: `package.json`
- Modify if generated: `Cargo.lock`

**Steps:**
1. Run `source ~/.cargo/env; ./scripts/bump-version.sh 0.11.0 --no-commit`.
2. Verify all package metadata reports `0.11.0`.
3. Check whether `Cargo.lock` changed and include it if needed.

### Task 3: Changelog And Release Notes

**Files:**
- Modify: `CHANGELOG.md`
- Modify: `RELEASE_NOTES.md`

**Steps:**
1. Review commits and diff from `v0.10.0..HEAD`.
2. Add a curated Keep-a-Changelog `0.11.0` entry dated `2026-06-09`.
3. Include itemized notes for core, CLI, semantic packs, authority evaluation, FFI bindings, security, docs, and CI/release hardening.
4. Regenerate or update `RELEASE_NOTES.md` from the curated changelog section.

### Task 4: Validation And Artifacts

**Files:**
- Create/modify: `dist/` artifacts if local build succeeds.

**Steps:**
1. Run release prechecks with tests.
2. Run cross-language tests through `just all-tests`.
3. Build release artifacts with `scripts/build-release.sh`.
4. Record any environment gaps, especially missing `maturin`, `wasm-pack`, or JavaScript package managers.

### Task 5: Commit, Tag, And Publish

**Files:**
- Commit release prep files.
- Create annotated tag `v0.11.0`.

**Steps:**
1. Commit release prep as `chore: release v0.11.0`.
2. Create annotated tag `v0.11.0` using release notes.
3. Push branch and tag.
4. Create a GitHub Release for `v0.11.0`.
5. Verify release build and publish workflows start and complete.

### Task 6: Post-Release Verification

**Steps:**
1. Confirm GitHub release URL.
2. Confirm crates.io, PyPI, npm napi, and npm WASM versions are available or identify workflow failures.
3. Update `.agents/current_state.md` and `.agents/next_steps.md` with evidence.
